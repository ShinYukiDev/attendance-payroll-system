# Research: 勤怠入力画面

## Decision 1: UI は Blazor フォーム検証 + サーバー検証 + DB 制約の三層で実装する

- Rationale: 入力時の即時フィードバックと、保存時の業務ルール強制、永続層での整合性確保を両立できる。
- Alternatives considered:
  - UI 検証のみ: API 直呼び出しや同時更新で破綻しやすく不採用。
  - サーバー検証のみ: 入力体験が低下するため不採用。
  - DB 制約のみ: エラーメッセージの業務的分かりやすさが不足するため不採用。

## Decision 2: CRUD 後の一覧反映は局所更新を基本とし、競合時のみ再読込する

- Rationale: 最小差分で体感性能を維持し、FR-014 の競合時再読込要件にも適合できる。
- Alternatives considered:
  - 毎回フルリロード: 単純だが操作継続性と性能が悪化するため不採用。
  - クライアント楽観更新のみ: サーバー確定値との不整合リスクがあるため不採用。

## Decision 3: 同時更新競合は EF Core の ConcurrencyToken で検出し、保存を拒否する

- Rationale: SQLite でも実装可能な楽観ロック方式であり、要件どおり後続更新を拒否できる。
- Alternatives considered:
  - Last Write Wins: 変更喪失を招くため不採用。
  - 悲観ロック: 運用コストと待ち時間増大のため不採用。
  - 全列比較: 保守負荷が高いため不採用。

## Decision 4: 監査ログは専用テーブルを作成し、3か月保持を日次ジョブで担保する

- Rationale: 登録・更新・削除の追跡要件を満たしつつ、保持期間ポリシーを明確に運用できる。
- Alternatives considered:
  - アプリ通常ログへ出力のみ: 検索性・保持管理が弱く不採用。
  - DB トリガーのみ: 操作主体情報との紐付けが難しいため不採用。

## Decision 5: 監査ログの個人情報は最小化し、必要最小限の識別情報のみ保持する

- Rationale: 機微情報保護の憲章要件に従い、ログ漏えい時の影響を抑制できる。
- Alternatives considered:
  - 変更前後の全値保存: 過剰取得でリスクが高く不採用。
  - 完全匿名化: 追跡要件を満たせないため不採用。

## Decision 6: テストは xUnit で先行定義し、業務ロジックの回帰防止を優先する

- Rationale: 憲章のテスト先行原則を満たし、実働時間計算・検証・競合ハンドリングの再発防止が可能。
- Alternatives considered:
  - 手動テストのみ: 回帰検知が遅く不採用。
  - 統合テスト中心: 初期段階では過剰なため不採用。

## Decision 7: ターゲットフレームワークは現行 net10.0 を維持する

- Rationale: 既存プロジェクト実態に合わせることで、機能外の基盤移行を避け最小差分を守る。
- Alternatives considered:
  - 直ちに .NET 8 へ移行: 影響範囲が大きく、今回機能の計画境界を超えるため不採用。
